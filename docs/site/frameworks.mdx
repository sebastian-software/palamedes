# Cross-Framework Story

> Internal messaging note. The published site lives in `site/`; this document
> records positioning language for maintainers.

The strongest thing about Palamedes is not any single adapter. It is that the
same i18n habits keep working as the app moves between framework families.

## The Claim

Palamedes keeps the parts teams feel every day stable while the host framework
changes:

- authoring stays macro-oriented and source-string-first
- runtime access stays centered on `getI18n()`
- message identity stays centered on `message + context`
- catalog QA stays in `ferrocat`
- adapters stay small instead of re-owning semantics

## Why This Matters

Teams do not want every framework decision to reopen their translation
strategy.

The more a team spans:

- multiple products
- a migration from one framework to another
- SSR and client-heavy surfaces
- future translation workflow requirements

the more valuable a stable i18n foundation becomes.

## Verified Framework Families

The current repo verifies Palamedes across:

- Next.js
- TanStack Start
- SolidStart
- Waku
- React Router

That matters because these are not small variants of one setup. They exercise
different server, routing, and rendering assumptions while preserving the same
Palamedes model.

## The Real Competitive Contrast

Many i18n tools are strong in one framework, one runtime style, or one product
shape.

Palamedes is unusual because it combines:

- framework breadth
- one runtime model
- one identity model
- one catalog workflow
- visible verification instead of vague compatibility language

That is the cross-framework story to repeat.
