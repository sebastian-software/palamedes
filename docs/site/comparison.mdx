# Comparison Messaging

Palamedes should be framed by product shape, not by a shallow feature checklist.

## Approved Comparison Line

Palamedes is for teams that like compile-time authoring and want a cleaner,
more coherent architecture underneath it.

## Lingui

Lingui is the closest direct comparison.

Use this framing:

- familiar authoring feel
- cleaner runtime and identity end state in Palamedes
- thinner adapter boundary
- narrower, more opinionated architecture

Do not frame Lingui as obsolete. Frame Palamedes as a sharper default for teams
already aligned with the same core instinct.

## next-intl

`next-intl` should be positioned as a stronger fit for teams that want a
Next.js-native, message-file-first center of gravity.

Use this framing:

- different default mental model
- some convergence on compile-time concerns
- not a clean benchmark peer to Palamedes end-to-end

## General Translation

General Translation should be framed as broader in product scope.

Use this framing:

- not just an i18n runtime/tooling library
- includes wider translation workflow and service-oriented concerns
- useful adjacent signal, but a different category boundary

## Safe Repetition Lines

- "Palamedes is narrower than some alternatives on purpose."
- "The differentiator is not just speed. It is a cleaner end state."
- "The architecture stays coherent across framework boundaries."
- "The repo shows proof, not just positioning."
